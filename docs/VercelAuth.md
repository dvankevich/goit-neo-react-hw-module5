Щоб Vercel CLI міг "протягнути" ваші змінні та правильно налаштувати локальне середовище, він має бути авторизованим та зв'язаним з вашим акаунтом.

---

## 1. Авторизація та зв'язок з проєктом (Первинне налаштування)

Перед тим як запускати `vercel dev`, необхідно один раз пройти процес "логіну" та зв'язати локальну папку з хмарним профайлом Vercel.

1. **Вхід в акаунт**:
Виконайте команду у вашому терміналі:
```bash
vercel login

```


Виберіть спосіб входу (GitHub, GitLab або Email). CLI відкриє браузер для підтвердження. Після успішного входу поверніться в термінал.
2. **Зв'язування проєкту (Linking)**:
Перебуваючи в кореневій папці вашого проєкту, виконайте:
```bash
vercel link

```


CLI поставить кілька запитань:
* *Set up and deploy?* — **Yes**
* *Which scope?* — Виберіть свій акаунт.
* *Link to existing project?* — Якщо ви вже створили проєкт на сайті Vercel, виберіть **Yes**, якщо ні — **No** (CLI створить новий проєкт у вашому кабінеті).



---

## 2. Робота зі змінними оточення через CLI

Замість того, щоб щоразу писати `export TMDB_TOKEN=...`, найкраще використовувати вбудовані можливості синхронізації Vercel.

1. **Синхронізація змінних з хмари**:
Якщо ви вже додали токен у панелі керування Vercel (на сайті), ви можете просто "витягнути" його локально:
```bash
vercel env pull .env.local

```


Ця команда створить (або оновить) файл `.env.local`, додавши туди всі змінні, які ви прописали в налаштуваннях проєкту на сайті. Це найнадійніший спосіб переконатися, що назви змінних співпадають.
2. **Додавання змінної через термінал**:
Ви також можете додати змінну прямо з командного рядка:
```bash
vercel env add TMDB_TOKEN

```


CLI запитає значення токена і в яких середовищах його використовувати (Production, Preview, Development). Виберіть усі.

---

## 3. Локальний запуск після авторизації

Тепер, коли проєкт "залікований" (`linked`) і змінні підтягнуті, запуск стає максимально простим:

```bash
vercel dev

```

**Чому це важливо:**
Коли ви робите `vercel link`, CLI створює у вашому проєкті приховану папку `.vercel`. У ній зберігаються `projectId` та `orgId`. Це дозволяє `vercel dev` точно знати, які саме серверні функції та налаштування використовувати, щоб вони на 100% відповідали "бойовому" серверу.

---

### Оновлений план дій (Checklist):

1. `npm install -g vercel` (встановили).
2. `vercel login` (авторизувалися).
3. `vercel link` (зв'язали папку з акаунтом).
4. `vercel env pull .env.local` (підтягнули токени).
5. `vercel dev` (працюємо локально).
